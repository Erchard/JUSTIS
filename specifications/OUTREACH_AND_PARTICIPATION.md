# Стратегія користі, публічності й участі

## 1. Для чого проект може бути корисним

Проект може бути корисним як:

- короткий Кодекс базових принципів справедливості;
- прозора база аргументів і контраргументів;
- освітній інструмент для морального мислення;
- інструмент критики законів, політик і інституцій;
- міжкультурний майданчик для пошуку спільного етичного ядра;
- структурований корпус для нейромережевих агентів;
- пам'ять про те, чому принцип було прийнято і коли його треба переглядати.

## 2. Кому проект може бути корисним

- людям, які шукають моральну орієнтацію;
- юристам, філософам, богословам, психологам, освітянам;
- правозахисникам і громадським організаціям;
- авторам політик і внутрішніх правил;
- дослідникам штучного інтелекту й етики;
- нейромережевим агентам, які потребують структурованого корпусу аргументів.

## 3. Принцип публічності

Публічність проекту має бути не рекламою готової істини, а запрошенням до перевірки.

Кожна публічна теза має по можливості вести до:

- короткої формули Кодексу;
- повного протоколу дискусії;
- дерева аргументів;
- відкритих заперечень;
- способу подати контраргумент.

## 4. Рівні участі

Участь має бути багаторівневою.

Людина або агент може:

1. поставити питання;
2. запропонувати приклад несправедливості;
3. додати джерело;
4. сформулювати аргумент;
5. сформулювати контраргумент;
6. вказати на приховане упередження;
7. запропонувати зміну формули закону;
8. підготувати дерево аргументів;
9. підготувати повний протокол дискусії.

## 5. Канали залучення

Початкові канали:

- GitHub repository;
- GitHub issues для відкритих питань;
- pull requests для структурованих внесків;
- короткі публікації в соціальних мережах;
- окремі пости для кожного закону Кодексу;
- agent-readable документи `AGENTS.md`, JML/YAML і реєстри.

## 6. Соціальні мережі для агентів

Спеціальних стабільних "соціальних мереж для нейромережевих агентів" може не бути як загального стандарту.

Практичний еквівалент на поточному етапі:

- публічний Git-репозиторій;
- машиночитані JML/YAML-блоки;
- чіткі задачі в issues;
- короткий `AGENTS.md`;
- шаблони внесків;
- позначення рівня джерельної надійності;
- окремий статус для агентських пропозицій, які ще не перевірені людиною або парламентом.

## 7. Правила для агентських внесків

Агент може:

- шукати прогалини;
- пропонувати джерела;
- реконструювати позиції;
- збирати контраргументи;
- перевіряти внутрішні суперечності;
- пропонувати JML-структуру;
- готувати чернетки протоколів.

Агент не може самостійно:

- оголошувати закон Кодексу остаточним;
- приховувати рівень джерельної надійності;
- змішувати цитату з реконструкцією;
- видавати симуляцію дискусії за повну ратифікацію без протоколу;
- закривати суттєве заперечення без відповіді.

## 8. Мінімальний публічний пакет

Перед активним поширенням проекту бажано мати:

- оновлений `README.md`;
- `CONTRIBUTING.md`;
- `AGENTS.md`;
- перший закон Кодексу;
- принаймні один повний протокол дискусії;
- реєстр відкритих питань;
- зрозумілий опис, як подати аргумент або контраргумент.

## 9. Ризики

Основні ризики:

- публічність стане маркетингом замість пошуку істини;
- кількість внесків знизить якість;
- найгучніші голоси витіснять вразливі;
- агенти створять багато правдоподібного, але слабко перевіреного тексту;
- проект стане надто складним для нових учасників.

## 10. Запобіжники

- кожна теза має посилатися на протокол або питання;
- кожне джерело має рівень надійності;
- кожне заперечення має статус;
- агентські внески мають бути позначені;
- прості форми участі мають бути дозволені;
- рішення Кодексу проходять через Небесний парламент і реєстри.

## 11. Найближчі дії

1. Створити `CONTRIBUTING.md`.
2. Створити `AGENTS.md`.
3. Оновити `README.md`.
4. Створити GitHub issue templates у майбутньому.
5. Підготувати короткий публічний опис першого закону.
